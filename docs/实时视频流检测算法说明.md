
## 取帧时机

  

后端在应用启动后持续读取 RTSP 流，每 **120ms** 取一帧，与上一帧做差值比较，判断画面是否发生变化。

  

---

  

## 何时将帧送给 VLM

  

### 1. 运动触发（后台自动）

  

每次取到预览帧后，将当前帧与上一帧做运动检测：

  

- 把两帧都缩小到 **32×32** 灰度图

- 逐像素求绝对差值，差值 > **18** 的像素计为"变化像素"

- 变化像素占比 ≥ **3%** 判定为检测到运动

  

检测到运动时，如果上一次的 VLM 调用已经完成，则立即将当前帧发给 VLM；若 VLM 仍在处理上一帧则跳过（不排队，不堆积）。第一帧始终发送。

  

### 2. 用户发起会话（单次）

  

用户发送文字消息时，后端从内存取出当前最新帧，与用户的问题一起发给 VLM，**仅取一次**，不做连续采集。

  

---

  

## 运动帧送给 VLM 后做什么

  

后端在 `prompt.py` 中预定义了若干**场景**，每个场景包含一个 `id`、名称和检测标准（criteria）。运动帧到达时，**单次** VLM 调用完成所有场景的识别：`build_detection_prompt()` 将所有场景拼入同一个系统提示，VLM 输出最匹配的 `scene_id`，若均不匹配则输出 `null`。

  

VLM 对每帧返回固定格式的 JSON：

  

```json

{ "scene_id": "phone_pickup", "confidence": 0.82, "reason": "右侧人物已把桌上的手机拿到手中" }

```

  

均不匹配时：

  

```json

{ "scene_id": null, "confidence": 0, "reason": "无匹配场景" }

```

  

新增场景只需在 `SCENES` 列表中追加一个 `Scene(id, name, criteria)` 对象，无需改动其他代码。

  

---

  

## 事件判定逻辑

  

每个场景独立维护状态，后端按以下规则决定是否触发告警：

  

```

单帧 detected=true 且置信度 ≥ 0.6

↓ 是

命中场景

↓

距上次告警已超过 30 秒冷却期

↓ 全部满足

发送 Webhook，payload 包含 scene_id，进入冷却

```

  

任意一帧 `detected=false` 或置信度不足，该场景的连续计数清零重新累积。

  

三个阈值通用于所有场景，可通过环境变量调整：

  

| 变量 | 默认值 | 含义 |

|------|--------|------|

| `PHONE_PICKUP_CONFIDENCE_THRESHOLD` | `0.6` | 单帧最低置信度 |

| `PHONE_PICKUP_CONSECUTIVE_HITS` | `1` | 触发告警所需连续命中帧数 |

| `PHONE_PICKUP_COOLDOWN_SECONDS` | `30` | 两次告警之间的最短间隔（秒） |

## 存在的问题
1.取帧间隔，自定义的120ms，如果取帧间隔时间过长，可能会丢失关键帧，如果取帧时间过短，可能造成VLM队列堵塞，造成VLM 处理的是几秒前的旧画面；
2.实现中采取丢帧策略，如果VLM忙就跳过新帧，实际测试中，即便丢帧，还是有两三秒的延迟，延迟在于关键帧前面的无效帧占据了处理队列，比如检测场景”拿起手机玩“，手开始移动就开始抽帧，但直到”拿起手机“这个关键帧才映射到定义的场景；
3.如果并发处理帧可能会造成乱序，比如第 10 帧先返回，第 8 帧后返回。后返回的旧帧可能覆盖新状态，导致系统基于过时画面触发。
